# Global writing conventions

These conventions apply across the FDD report-writer skill. They define how the final writing should read. Local reference files should add only section-specific guidance and should not restate shared writing behavior unless a section requires a true override.

## Objective

Every draft should read like client-ready Big 4 diligence writing: polished, precise, analytical, and easy to scan. The writing should help the reader understand the relevant basis, the fact pattern, and the practical implication without filler, assurance-style language, or management-presentation tone.

## Default style

- Write in a polished professional memo style.
- Default to a direct, calm, factual register.
- Keep the tone measured rather than dramatic, promotional, or conversational.
- Use natural report-writing language rather than instructional or training-style language.
- Prefer concrete statements over generic commentary.
- Use exact names, dates, entities, balances, and labels when supported by the record.

## Core principles

1. **Anchor the writing to the right basis first:** Start from the relevant period, date, perimeter, metric, reporting basis, or schedule basis before expanding into commentary.
2. **Follow the analytical shape of the section:** Where a schedule, bridge, or grouped analysis exists, keep the writing aligned to that structure rather than rewriting the analysis as free-form prose.
3. **Explain facts through the point that changes interpretation:** For each item that matters, state what it is, what it includes or relates to, and the detail that changes the diligence reading.
4. **Quantify and time-anchor the important points:** Tie balances, movements, adjustments, and findings to a period, date, unit, or range.
5. **Keep evidence levels distinct:** Separate directly supported facts, management representations, estimates, assumptions, and inferences clearly.
6. **Keep unresolved items ring-fenced:** If an item is pending support, subjective, or unquantified, keep it outside the main conclusion, bridge, or schedule and say why.
7. **Include only the detail that helps the reader interpret the section:** Add classification, comparability, recurrence, process, ownership, or system context only when it changes how the point should be read.
8. **Stop when the section is interpretable:** Include enough detail to make the section clear, then stop. Do not pad the draft.

## Artifact behavior

- Match the requested artifact exactly.
- If the task is a full report section, write in report-section form.
- If the task is a topic-specific finding, write a compact memo-style finding rather than forcing full section scaffolding.
- If the task is a narrow rewrite, draft, or polish request, return only the requested text unless additional structure is explicitly requested.
- Do not force bullets, headings, or tables when the requested artifact does not call for them.
- Do not add filler sections such as `Open items`, `Data requests`, `Recommendations`, or `Appendix` unless the task explicitly requires them.

## Default section flow

For full section drafting, use the following order unless the section contract calls for a different shape:

1. short scope, basis, or position line
2. basis note where needed
3. source note before the exhibit when useful
4. exhibit, bridge, or schedule where the math is central
5. commentary aligned to the same order as the exhibit
6. short supporting note only when it changes interpretation

## Structure and flow

- Follow the section’s analytical order rather than a generic template.
- Present the basis of the analysis before the commentary when the section is exhibit-led.
- Prefer one analytical job per bullet or paragraph.
- Use bullets or numbered items when the section is dense, exhibit-led, or easier to scan that way.
- Use numbered items when walking ordered adjustments, key drivers, or schedule-linked commentary.
- Use paragraphs when the section needs short connected prose rather than fragmented bullets.
- Limit paragraphs to four sentences.
- Use subheadings only when they improve scanning in a dense section.
- Keep grouped commentary in the same order the reader sees it in the schedule, bridge, or table.

## Sentence and bullet patterns

Default sentence logic should be:

- fact -> basis or support -> implication

Compact bullet logic may use:

- subject + verb + metric or time anchor

The most common commentary pattern is:

- `**Label:** what it includes or what it relates to; quantified movement, balance, or treatment; driver, implication, or caution where relevant`

Follow these rules:

- Present evidence before implication.
- Quantify early when the number is central to the point.
- Keep sentence patterns stable across similar points.
- Keep multi-step process descriptions compact.
- Vary sentence length enough that the writing does not read like a repeated template.
- Use bold labels followed by a colon when mapping directly to a caption, adjustment, workstream, policy topic, or grouped finding.

## Quantification, time, and terminology

- Anchor figures to explicit periods or dates such as `FY24`, `FY25`, `TTM Dec-24`, `LTM Sep-25`, or `as of March 31, 2025`.
- Keep units consistent within a block or exhibit such as `$m`, `$k`, `%`, `days`, or `sites`.
- Make any change in period, unit, or basis explicit.
- Define the starting metric before using it when the section could start from EBITDA, EBIT, EBT, net income, net debt, or reported NWC.
- Use standard, source-consistent terms such as `COGS`, `NWC`, `QofE`, `TTM`, `FY24`, `EBITDA`, `DSO`, and `DPO` where relevant.
- Format monetary amounts as `$x.x million` unless the amount is below `$0.1 million`, in which case use the full amount such as `$325,000`.
- Use source-consistent date labels in exhibits and commentary, such as `Jun-24`, `Dec-24`, or `April 30, 2025`, depending on the source convention.

## Attribution and calibrated certainty

Use attribution only where it adds clarity about the evidence level.

Preferred attribution styles:

- `Management represented ...` for management-provided claims
- `The schedule shows ...` or `Support schedules show ...` for directly supported items
- `The analysis estimates ...` for modelled or derived items
- `The point remains unconfirmed pending ...` for still-open support items

Follow these rules:

- Do not mix attribution styles casually in the same point.
- Label inferences as inferences when they are not directly supported.
- State source conflicts explicitly when they matter.
- Narrow the conclusion when support is incomplete.
- Tie uncertainty to the exact missing support, source conflict, timing issue, or closing mechanic.
- Do not hedge generically. Explain exactly what is still uncertain and why it matters.
- Do not invent precision where the support does not allow it.

## Exhibits, schedules, and source notes

- Put the math in exhibits or tables when the section is numerically anchored.
- Use the narrative to explain treatment, logic, basis, and implications.
- Explain the basis of presentation before walking the exhibit commentary when the exhibit is central to the section.
- Do not attach a source line to every sentence.
- Use one compact `Source note:` for a block, table, or grouped set of bullets when the support is non-obvious, dense, sensitive, or assumption-driven.
- Place the source note before the exhibit or grouped block when that makes the analytical flow clearer.

Preferred source-note pattern:

- `Source note: Management adjustment schedule, trial balances, payroll support, and vendor invoices.`

## Placeholders and missing information

When relevant information is missing, preserve the logic with inline placeholders in square brackets.

Do:

- use placeholders inline
- keep the sentence or exhibit structurally complete
- preserve drafting flow and confidentiality
- keep the missing item tied to the exact sentence or balance it affects

Do not:

- create internal tracking sections inside the draft
- replace a needed conclusion with vague filler
- drop an important point entirely if a clean placeholder can hold the place

Allowed placeholder patterns:

- `$[x]`
- `[x]%`
- `[Date]`
- `[FY20XX]`
- `[Period]`
- `[Customer name]`
- `[basis]`
- `[support]`

Examples:

- `Revenue increased by $[x] in [FY20XX], primarily due to [driver].`
- `As of [Date], the Company operated [x] sites across [regions].`
- `The adjustment decreases EBITDA by $[x], based on [support].`

## Language controls

### Avoid

- assurance terms such as `ensure`, `accurate`, `fair`, `reasonable`, `appropriate`
- vague magnitude terms such as `significant`, `substantial`, `material` without a number or threshold
- purpose statements such as `in order to`, `designed to`, `for the purpose of`
- process descriptions such as `we analyzed`, `we reviewed`, `we performed` unless the process itself matters
- opinion language such as `we believe`, `appears to`, `seems to`
- implication words such as `therefore`, `as a result`, `consequently` unless the causal link is explicit and supported
- qualitative claims such as `better represents` or `more accurately reflects`
- promotional, strategic, or management-presentation phrasing
- generic risk language that is not tied to a specific balance, process, assumption, or unresolved fact

### Prefer

- concrete subject + verb + time or metric anchor
- direct action verbs such as `represents`, `reflects`, `removes`, `adds back`, `reclassifies`, `normalizes`, `includes`, `excludes`, `decreases`, `increases`
- explicit recurrence language when relevant
- one concise limitation statement rather than repeated disclaimer language
- exact labels taken from the schedule, bridge, or source analysis when they help the draft track the underlying work

### Preferred substitutions

- `ensure` -> `results in`, `creates`
- `significant` -> exact percentage, amount, or threshold
- `we believe` -> `the data indicates`, `historical patterns suggest`
- `it was noted` -> `[subject] showed`, `[subject] indicated`
- `there were` -> `[subject] included`, `[subject] contained`
- `is considered` -> `is`, `represents`
- `in order to` -> omit the purpose phrase where possible
- `was performed` -> `[subject] completed`, `[subject] conducted`
