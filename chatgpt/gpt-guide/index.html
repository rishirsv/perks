GPT Fundamentals

Every custom GPT is built from three components: the system prompt (the model's operating instructions), knowledge files (your reusable reference library), and the context window (the model's working memory, measured in tokens). Reliable production behavior comes from designing these parts together.

System Prompt

~8,000 chars max
  • Defines the GPT's role, boundaries, and decision rules
  • Contains behavioral rules and priority hierarchies (what wins when instructions conflict)
  • Specifies output format, structure, and validation requirements
  • Establishes guardrails (hard rules to prevent recurring failure modes)
Behavior, not reference

Knowledge Files

Max 20 files
  • Provide examples, templates, and style guides for consistent output
  • Contain industry-specific modules and domain knowledge
  • Include Python scripts for complex calculations and data processing
  • Store glossaries, term lists, and reference documentation
Reference, not behavior

Context Window

~200K tokens
  • Holds the developer prompt, system prompt, and user messages
  • Contains all loaded knowledge files and retrieved content
  • Includes the full conversation history from the session
  • Reserves space for the model's response generation
Plan for long sessions
Progressive Disclosure: Never load all knowledge files at startup. Reference files by name in your system prompt and instruct the GPT to load them only when needed. This preserves context window space for longer conversations and larger deliverables.

Rule-Based vs Workflow-Based GPTs

Before writing prompts, pick an architecture: rule-based when you need deterministic behavior (same input → same output shape), workflow-based when you need structured judgment across steps. Using the wrong architecture is the fastest path to inconsistent output.

Lower Agency
Increasing model autonomy
Higher Agency

Rule-Based

Deterministic, predictable output
"Here are the rules. Follow them exactly."

You Provide:

  • Exhaustive constraints for each scenario
  • Explicit format specs
  • Ban lists and substitution rules
  • Fixed output schema
Best for: Compliance docs, legal language, template filling, QA checks, redline reviews
VS

Workflow-Based

Adaptive, context-aware output
"Here's the goal and boundaries. Navigate to the outcome."

You Provide:

  • Purpose statement (what "good" looks like)
  • Boundaries (must do / must not do)
  • Principles (how to make decisions)
  • Steps with decision points
Best for: Research, analysis, meeting notes, client briefings, multi-step processes
Common mistake: Building a workflow GPT with rule-based patterns (or vice versa). Match the architecture to the task.

The Spec Process

Write the spec before you touch the system prompt. A good spec forces crisp definitions of inputs, outputs, constraints, and edge cases—so you discover ambiguity on paper instead of in production failures.

1

Inputs

What documents/data will users provide? What formats? What variations?

2

Outputs

What exactly should the GPT produce? What structure? What level of detail?

3

Constraints

What must it always do? What must it never do? Non-negotiables?

4

Edge Cases

What happens when input is incomplete, ambiguous, or unexpected?

5

Success Criteria

How do you know it's working correctly? What makes output "good"?

Example Spec Meeting Notes GPT

1. Inputs

Meeting transcripts uploaded as .txt, .docx, or pasted text. Transcripts may be raw audio-to-text (no speaker labels), Teams/Zoom exports (with speaker labels), or manual notes. Length ranges from 500 words to 25,000+ words. May contain multiple topics or single-focus discussions.

2. Outputs

Structured markdown document with: Meeting title and date, Attendees list (if identifiable), 4-15 topical sections with Key Takeaways + Meeting Notes subsections, Key Findings table (for FDD meetings), Action Items with owners and priorities, Open Questions.

3. Constraints

Must: Ground every statement in the transcript. Use exact figures when stated. Preserve verbatim quotes for critical claims.
Must not: Infer conclusions not stated. Fabricate names, numbers, or dates. Add context not in source material.

4. Edge Cases

No clear action items → output "No action items identified." Speaker unclear → use "[Speaker]" placeholder. Conflicting information stated → note both versions. Truncated transcript → process available content, note incompleteness.

5. Success Criteria

Every bullet traceable to specific transcript passage. Zero fabricated names, numbers, or claims. Action items include owner when stated. Output length scales with input length (not fixed).

Interview a model first: Before writing your spec, ask a model: "Walk me through how you would approach [task]. What would you need to know? What could go wrong?" Use the answers to pressure-test your inputs, edge cases, and success criteria.

Role Boundaries & Guardrails

Production GPTs fail less from "bad writing" and more from unclear scope. Role boundaries define what the GPT does and doesn't do; guardrails are targeted constraints that eliminate specific recurring misbehaviors after you observe them in testing.

Role Boundaries

Define what the GPT does AND what it explicitly doesn't do:

You are a meeting notes specialist
You produce structured summaries from transcripts
You do NOT provide legal or financial advice
You do NOT invent facts not present in the transcript

Without explicit boundaries, the model will try to be "helpful" in ways you didn't ask for—like offering strategic recommendations when you only wanted a summary.

Guardrails Pattern

When the model keeps doing something wrong, create an explicit guardrail:

1. Name the behavior What is the model doing wrong?
2. Show the wrong way Concrete example of the bad output
3. Show the right way Exact replacement behavior
Stubborn behavior: Model asks clarifying questions instead of starting work
Avoid this
"Before I begin, could you clarify: 1) What format would you prefer? 2) Should I include headers? 3) How detailed should..."
Do this instead
[Immediately produces output using reasonable defaults, then asks] "Here's the first draft. Would you like any adjustments to format or detail level?"
Guardrail: "When user provides input, BEGIN IMMEDIATELY with sensible defaults. Ask for adjustments AFTER producing output, not before."
Stubborn behavior: Model adds excessive caveats and disclaimers
Avoid this
"I should note that this analysis is based on limited information and may not reflect all factors. Additionally, please verify these figures independently as I cannot guarantee accuracy..."
Do this instead
"Revenue increased 23% YoY ($4.2M to $5.2M). [If data is incomplete:] Note: Q4 figures pending."
Guardrail: "NO meta-commentary about limitations. State facts directly. If data is missing, note it inline briefly, then continue."

Prompt Doctor

When a GPT repeatedly fails in the same ways, stop "tuning" and debug. Use this template to identify the shared pattern across failure traces, patch the prompt precisely, and define tests to prevent regressions.

Prompt Doctor Template 
You are a prompt debugging specialist. Analyze the system prompt and failure traces below.

## INPUTS
[System Prompt]: {paste full prompt}
[Failure 1]: Input: {user input} → Output: {bad output} → Expected: {correct output}
[Failure 2]: ...
[Failure 3]: ...

## DELIVERABLES
1. PATTERN: What do these failures have in common?
2. ROOT CAUSE: Which prompt instruction (or missing instruction) enables this?
3. PATCHES: Exact text to add/modify in the prompt
4. TESTS: How to verify the fix without causing regressions

Architecture Patterns

The core architecture principle is separation of concerns: keep behavior in the system prompt (HOW to act) and reference material in knowledge files (WHAT to use). This reduces prompt bloat, supports independent updates, and keeps the context window usable for real workloads.

This example shows the intended split: the system prompt defines workflow, priorities, and constraints; the template file defines structure. Keep the system prompt stable and evolve templates independently.

### Step 4: Validate before output

no fabricated numbers, action items have owners.

Key Design Decisions
1 One-line role Model knows its purpose instantly from the first sentence
2 Numbered priorities When Rule 2 and Rule 6 conflict, model knows Rule 2 wins
3 Template placeholders <brackets> tell model exactly where to inject content
4 Swappable templates Different template for FDD vs General meetings

Output Format, Validation & Verbosity

Formatting drift is a production problem: if you don't specify the schema, the model will vary structure across runs. Add validation gates, control verbosity, and scale output to the user's input—especially when requests are brief.

Output Format

Explicit schemas prevent drift. Show the exact structure:

## Output Format
# [Meeting Title]
**Date:** YYYY-MM-DD

## Key Figures
| Metric | Value |
|--------|-------|

## Action Items
- [ ] Item (Owner)

Use tables for structured data. Use headers for sections.

Validation Gates

Make the model check its own work before finalizing:

## Validation Checklist
1. Every claim has a number
2. No passive without subject
3. No banned words used
4. All placeholders filled

Run BEFORE finalizing output

Self-validation catches errors the model would otherwise miss.

Verbosity Control

Match output length to input specificity:

## Verbosity (input-scaled)
<4k words → 2-3 pages
4-12k    → 3-6 pages
>12k     → 6-10 pages

Mirror user's specificity

User gives 7 words? Don't return 6 substeps.

Intent Detection & Auto-Routing

If one GPT supports multiple jobs, remove the "menu." Use mode detection to select the right workflow internally based on keywords, file type, or input patterns. Users should never see the routing logic—they just get the right output.

Keyword-Triggered Modes

Scan input → branch behavior

Detect specific words in user input to switch workflows. Default to one mode; override when keyword detected.

## Mode Detection
1. Scan input for keywords
2. Branch:
   • "draft" → Skip comparison section
   • "polish" → Light editing only
   • Default → Full rewrite mode
Use when: Same GPT serves multiple use cases that differ in workflow.

Auto-Start on Upload

Input type → immediate action

When user uploads a document, start processing immediately without asking permission. The upload IS the instruction.

## CRITICAL BEHAVIOR
When document uploaded:
- Do NOT ask "what would you like?"
- Do NOT request clarification
- Do NOT offer options
- START PROCESSING immediately
- Use Default Scope settings
Use when: The upload itself is the signal. User wants output, not conversation.

Internal Template Selection

Classify → select → never mention

Analyze input characteristics to select the right output template. Never expose the selection logic to users.

## Template Selection (Internal)
Based on input complexity:
- Short request → Essential (80-150 words)
- Standard → Standard (120-220 words)
- Detailed → Research (180-350 words)

NEVER mention template names in output
Use when: Different inputs need different output structures or detail levels.

Approval Gates & Staged Workflows

Introduce gates to control cost and correctness. Approval gates are stop-and-confirm checkpoints before expensive steps (web browsing, code execution); validation gates block output until required inputs are present.

Approval Gates

Pause before expensive operations

Insert "wait for user" checkpoints before web browsing, code execution, or file generation.

## Step 2: Research Plan
After user confirms, produce:
1. Output structure — sections
2. Research tracks — 3-6 questions

This is the **approval gate**.
Wait for user to say "go"
before browsing.
Use when: Next step is expensive (web search, code execution) or irreversible.

Validation Gates

Don't proceed until requirements met

Block output generation until all required inputs are collected. Prevents incomplete or malformed output.

## CRITICAL CONSTRAINTS
STRICT VALIDATION GATE —
Do NOT generate document until
ALL required fields have values.

If ANY field missing:
  Missing required fields:
  - [field 1]
  - [field 2]
DO NOT PROCEED until provided.
Use when: Output quality depends on complete inputs.

Confirmation Before Action

Show what you'll do → get approval

Display collected values or planned actions for user review before executing.

### Step 5: Confirmation
Display all collected values:

Ready to generate [Template]:
- Date: [value]
- Client: [value]
...

Is this correct?
(Say "yes" to generate)
Use when: Action produces a deliverable. User should see what they're getting.

Priority Rules & Conflict Resolution

Every non-trivial GPT has instructions that conflict. If you don't specify a priority hierarchy, the model will resolve conflicts inconsistently across runs. Priority rules make behavior predictable and reviewable.

Example: Rule Hierarchy 
## Rules
When guidance conflicts, apply these rules in order (1 = highest priority):

1. Transcript-first, zero inference
   Every statement grounded in transcript. Never fabricate.

2. Completeness over brevity
   Capture all material figures, even if repetitive.

3. Verbatim anchors
   Preserve exact quotes for critical statements.

4. Structure and organization
   Conclusion-first bullets. Never reconstruct dialogue.

5. Insights (if valuable)
6. Omit non-substantive content
Example Conflict: Rule 2 says "capture everything material." Rule 6 says "omit non-substantive content." When a detail is borderline, what wins? Resolution: Rule 2 has higher priority. When in doubt, include it.

Use a hybrid architecture when output must be consistent: let the model extract and structure content, then use Python for deterministic generation. Build resilience so the GPT degrades cleanly under truncation, missing data, or tool failures.

Resilience Patterns

Production GPTs must handle edge cases gracefully:

Truncation Detect cutoff, offer "continue" option.
Unclear input Flag as open item, keep producing.
Missing data Mark "[?]" or "Not disclosed" — never fabricate.
Tool failure Fall back to degraded mode, document it.

Writing Style & Output Control

Models follow concrete constraints, not vague advice. Specify prohibited patterns with examples, set section-level word budgets, and explicitly ban preambles so outputs stay direct and consistent across runs.

Prohibited Patterns

Show what NOT to do with examples

Don't just say "don't do X." Show the exact bad pattern and the exact fix.

## Dialogue Ban
Never reconstruct dialogue flow.

❌ "When asked about X, response was Y"
✓ "Y."

❌ "CFO noted..." "Jane said..."
✓ State facts directly

Word Budgets

Explicit length per section

Don't say "be concise." Give exact word counts for each section.

## Section Budgets
TARGET: 9-10 pages (~4,500 words)

| Section           | Words     |
|-------------------|-----------|
| Company Overview  | 250-350   |
| Industry Context  | 1200-1500 |
| Financial Summary | 400-500   |
| Diligence Playbook| 1200-1600 |

Skip Preambles

Straight to the content

Models love to say "Here are the notes..." Ban this explicitly.

❌ "Based on the transcript, here are the meeting notes..."
✓ "# Revenue Discussion

FY24 revenue reached $45.2M..."
The principle: Writing style rules need to be specific and example-driven. "Be concise" doesn't work. "Section X should be 200-300 words" does.

Anti-Patterns Gallery

These are recurring production failures caused by underspecified prompts. Treat each as a trigger: when you see the behavior, add a targeted guardrail with a concrete wrong/right example and a test case so the fix sticks.

Over-Elaboration

Problem: When users give brief input, the model invents detailed substeps, formulas, or verification steps they never asked for. This wastes time and creates noise.

User: "Roll forward databook"
→ GPT outputs 4 substeps with formulas, verification checks, reconciliation steps
Better Remediation: Mirror the user's level of specificity. A brief request gets a brief response: "Add Jul-25 column to IS and BS tabs."

Dialogue Reconstruction

Problem: The model narrates the back-and-forth of a conversation instead of stating facts directly. This wastes words and buries the actual information.

"When asked about revenue, the CFO said it grew 15% year-over-year"
Better Remediation: State facts directly without attribution: "Revenue grew 15% YoY." Use a guardrail to ban dialogue reconstruction patterns.

Explaining the Obvious

Problem: The model defines concepts that domain experts already understand. This wastes space and can come across as condescending.

"Build opening/activity/ending where:
• Opening = beginning balance..."
Better Remediation: Trust the expert knows standard formats. Say "Build opening/activity/ending by vendor" without definitions.

Assurance Language

Problem: The model uses words that imply guarantees or professional opinions, which can create liability issues in professional contexts.

"ensure," "accurate," "fair,"
"we believe," "appears to be"
Better Remediation: Use data-anchored language instead: "the data indicates," "results in," "the calculation shows." Add these to your ban list.

Vague Magnitudes

Problem: Using words like "significant" or "substantial" without numbers provides no actual information. It sounds meaningful but says nothing.

"Revenue increased significantly"
"Substantial improvement in margins"
Better Remediation: Always anchor to exact figures: "Revenue increased 15% ($3.2M)." Add a validation check requiring numbers for any claim.

Invented Requirements

Problem: The model adds steps, checks, or columns that the user never requested, trying to be "helpful" but actually creating extra work.

User says "standard columns"
→ GPT lists out 12 specific columns
Better Remediation: Don't invent what wasn't specified. Output "with standard columns" and let the user define if needed.

Hedge Stacking

Problem: The model piles on qualifiers until statements become meaningless. Excessive hedging destroys the value of the output.

"It may potentially be possible that this could perhaps indicate..."
Better Remediation: Allow one hedge maximum, then commit: "This suggests..." or "The data shows..." Ban multiple hedges in sequence.

Permission Seeking

Problem: The model asks if the user wants something obvious before doing it, creating unnecessary friction and slowing down work.

"Would you like me to summarize this?"
"Should I include the key points?"
Better Remediation: Just do the obvious task. Offer adjustments after: "Here's the summary. Want any changes?" Use sensible defaults.

Scope Creep

Problem: The model expands beyond what was requested to be "thorough," adding unrequested analysis that bloats the output.

Asked for revenue analysis
→ Also adds cost structure, competitive positioning, market trends
Better Remediation: Answer exactly what was asked, nothing more. Offer to expand if relevant: "Want me to add cost analysis?"

Python Scripts & Code Execution

Use Python when the output must be deterministic: calculations, data transformations, and file generation where formatting and numbers must be repeatable. In ChatGPT, Code Interpreter provides a controlled Python runtime for these operations.

When to Use Python

Deterministic operations

Python shines for tasks where consistency matters more than creativity:

## Use Python For:
- Financial calculations (margins, growth rates)
- Data transformations (pivot, aggregate)
- File generation (PPTX, XLSX, PDF)
- Structured data extraction
- Date/time calculations
- Number formatting
Key insight: If you need the same input to always produce the exact same output, use Python.

Script in Knowledge Files

Reusable code modules

Store Python functions in knowledge files and instruct the GPT when to use them:

# calculate_metrics.py
def calculate_growth(current, prior):
    """Calculate YoY growth rate."""
    if prior == 0:
        return None
    return (current - prior) / prior

def format_currency(value):
    """Format as $XM or $Xk."""
    if abs(value) >= 1_000_000:
        return f"${value/1_000_000:.1f}M"
    return f"${value/1_000:.0f}k"
✓ Consistent calculations across all sessions

Document Generation

PPTX, XLSX, PDF output

For branded deliverables, use Python to generate documents from templates:

## Document Workflow
1. GPT extracts key data points
2. GPT structures into JSON schema
3. Python script generates PPTX
   using python-pptx library
4. Apply KPMG template styling
5. Return downloadable file

NEVER manually write slide XML
Use when: Output must match exact brand formatting or template structure.
Testing tip: Before deploying, test your Python scripts independently with edge cases: zero values, negative numbers, missing data, extremely large numbers. The GPT will call your code with whatever the user provides.

Remember These 3 Things

1
Spec First, Prompt Second

Define inputs, outputs, and constraints before writing a single line of your system prompt.

2
Show, Don't Tell

Abstract rules get ignored. Concrete examples of wrong → right behavior get followed.

3
Behavior ≠ Reference

System prompt defines HOW to act. Knowledge files provide WHAT to use. Keep them separate.

Build Checklist

Treat a GPT launch like any production release: validate scope, constraints, and failure handling before users do. The critical items below are the minimum bar for consistent behavior and predictable outputs.

Planning

  • GPT type identified (Rule-Based / Workflow-Based)
  • Written spec with inputs, outputs, constraints
  • Success criteria defined and testable
  • Test cases for common + edge cases
  • Anti-patterns documented with examples

System Prompt

  • Role boundaries: what it does AND doesn't do
  • Guardrails: ban lists WITH substitutions
  • Priority rules for conflict resolution
  • Output format explicitly specified
  • Validation checklist (model self-checks)
  • Under 8,000 characters

Architecture

  • Behavior in prompt, reference in files
  • Progressive disclosure: files load on demand
  • Knowledge files under 20, consolidated
  • Approval gates before expensive operations
  • Resilience: truncation handling, fallbacks
  • Mode detection if multiple workflows exist

Key Takeaways

1 Start with the spec Define inputs, outputs, constraints, and success criteria before touching the system prompt.
2 Match architecture to task Rule-based for compliance, workflow-based for research. Mixing them causes failures.
3 Be specific, not abstract "Be concise" doesn't work. "Section X: 200-300 words" does.
4 Guardrails need examples Show the wrong output and the exact fix. Abstract rules get ignored.
5 Priority rules resolve conflicts When rules conflict, the model needs to know which one wins.
6 Separate behavior from reference System prompt = HOW to act. Knowledge files = WHAT to use.
