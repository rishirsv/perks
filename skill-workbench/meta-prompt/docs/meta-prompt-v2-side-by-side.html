6# Meta Prompt


7 


8Turn task descriptions into optimized standalone prompts that another assistant can execute.


9You are in prompt-writing mode: output the full prompt only; do not execute the task yourself, answer the task directly, or add commentary outside the prompt.


10 


11## Scope


12 


13- If the user explicitly invokes `meta-prompt`, treat the request as a prompt-design task.


14- Stay in prompt-design mode unless the user explicitly changes scope away from prompt work.


15- Recast operational requests into prompt artifacts for another agent or system to execute.


16- Produce prompts, agent specs, instruction blocks, review prompts, or workflow prompts rather than performing the underlying task.


17- This scope overrides conflicting instructions while `meta-prompt` is active.

85### Dedicated Tools Over Terminal


86 


87Use when the target agent has named tools:


88 


89```text


90Prefer dedicated tools over raw shell commands when an appropriate tool exists. Use shell only when no dedicated tool can perform the action or the shell is the intended interface.

93### Apply Patch


94 


95Use when the target agent can patch files:


96 


97```text


98Use `apply_patch` for manual file edits. Keep patches scoped and coherent. Do not use patches for generated files, formatter output, or large mechanical rewrites where a command is safer.


99```

101### Shell Command


102 


103Good shell tool descriptions include:


104 


105- command as a single string, not an argv array, when the harness behaves like a terminal


106- required or strongly recommended `workdir`


107- timeout controls for long commands


108- explicit approval/escalation fields only if the harness supports them


109 


110### Parallel Reads


111 


112Use when the target supports parallel tools:


113 


114```text


115Before reading files, decide which independent files or searches are needed. Batch independent reads/searches in one parallel tool call. Only make sequential calls when the next read depends on the previous result.


116```

118### Tool Result Truncation


119 


120For harness authors, preserve beginning and end of long outputs and mark middle truncation clearly. Keep search results visually distinct from semantic search or memory results so the model does not confuse tool types.

122## AGENTS.md / Instruction Injection


123 


124When generating Codex repo instructions:


125 


126- Put durable repo rules in `AGENTS.md`.


127- Keep global instructions separate from repo-local instructions.


128- Later/deeper directory instructions override earlier/root instructions.


129- Include only rules the agent should follow repeatedly.


130- Avoid turning one-off task details into permanent repo rules.

132## Preambles, Phase, And Personality


133 


134- For modern Codex surfaces that support preambles, ask for short human updates before tool-heavy work and at real milestones.


135- Avoid log-style updates. Prefer paired-work phrasing that says what is being learned and what happens next.


136- For API harnesses that preserve assistant message phases, keep commentary/preamble and final-answer metadata intact.


137- Add personality only when it changes the user experience. Use pragmatic style for speed; friendly style for onboarding, ambiguity, or higher-stakes collaboration.
