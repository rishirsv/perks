# Development Mode

Use development mode when the user has one rough idea and wants to shape it into a clearer product, UX, architecture, or implementation direction.

## Start

Ask what the user is already thinking before offering your own complete solution, unless they explicitly ask you to lead.

Clarify only what changes the answer:

- problem and user
- trigger or entry point
- core flow
- state, data, and system behavior
- constraints and non-goals
- edge cases and failure modes
- success criteria or verification

Ask one focused question at a time. Offer a recommended default when useful.

## Ground Repo Work

For repo/software work, read `repo-grounding.md`. Inspect adjacent code and docs before asking questions that the repo can answer.

Keep implementation details out unless the user is explicitly shaping architecture or technical behavior.

## Explore Shapes

When real alternatives remain, present 2-3 concrete approaches.

For each approach include:

- brief description
- pros
- cons
- key risk or unknown
- when it is best suited

Do not invent filler options when one path is clearly better. State the recommendation directly and invite pushback.

## Shape UX And Implementation

For product or app ideas, cover:

- primary user flow
- entry points
- empty/loading/error/success states
- information hierarchy
- interaction model
- data or ownership implications
- verification or preview needs

Stop before turning this into an implementation plan. The output should be ready to hand to planning, not a plan itself.

## Output

End with:

- current best shape
- core flow
- key states
- ownership/data implications
- tradeoffs
- open questions
- recommended next step
